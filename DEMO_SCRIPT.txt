[1:00 — ARCHITECTURE]
(show assets/architecture_diagram.png)
Let us walk through the workflow architecture. When a user submits a cleanup request, it first passes through our Security Checkpoint node. If the query is clear of PII and malicious patterns, it goes to the central Orchestrator agent. (pause)
The Orchestrator acts as the brain, using AgentTools to delegate tasks to specialized sub-agents: the File Cleaner Agent and the Email Cleaner Agent. These sub-agents communicate directly with our custom stdio MCP Server to list directory contents or format rules. (pause)
The resulting plan goes to a Router Node. If the plan contains safe, non-destructive file moves, it is auto-approved and sent to the Final Output. However, if the plan involves dangerous operations like file deletions or purges, the router node intercepts the flow and routes it to the Human Review node, pausing the workflow until the user explicitly typing approval. (pause)

Since the proposed actions contain the destructive keyword "wipe", the Router Node immediately intercepts the flow. The playground pauses and prompts me: "Human Review Required... Type yes to approve, or no to reject." The agent will not touch any files until I type "yes" to resume. (pause)

Finally, let us test our security firewall: "Organize C:\Windows\System32 and delete old system files." (pause)
The Security Checkpoint node detects that the path targets a blacklisted system directory. The request never even reaches the LLM, saving token quota and protecting the host machine. (pause)
